# Autonomous Bundle Split Analyzer & Route-based Code Splitting Guide

**Status:** ✅ Active  
**Triggers:** PR to `main`, push to `main`, weekly Tuesday 17:00 UTC  
**Fail condition:** Any route bundle exceeds thresholds (non-blocking advisory by default; can be made blocking via CI)  
**Telegram alerts:** On regressions; weekly digest summary

---

## Problem

Next.js apps can accumulate large initial chunks and route-level bundles as features grow. Without visibility, routes become heavy, slowing First Contentful Paint (FCP) and Largest Contentful Paint (LCP). Code splitting opportunities (dynamic imports) are easily missed.

## Solution

Static analyzer that:
- Parses Next.js build `stats/` JSON files (generated by `next build`)
- Computes gzipped size estimates per route chunk
- Identifies routes where initial chunk exceeds 300KB gzipped or any route chunk exceeds 250KB gzipped
- Recommends specific dynamic import patterns based on route name heuristics (dashboards → charts; maps → map libs; editors → rich-text modules)
- Posts PR comments with actionable suggestions
- Opens GitHub issues on main-branch regressions
- Weekly Telegram digest summarizing route health

---

## How It Works

1. Requires a Next.js build (`npm run build`) to generate stats in `.next/build/stats/`
2. Scans `app-*.json` and `page-*.json` stats files
3. For each route, aggregates chunk sizes and estimates gzip (25% compression)
4. Compares against thresholds:
   - Initial chunk > 300KB → warning (critical for LCP)
   - Any route chunk > 250KB → warning
5. Generates per-route recommendations using common patterns:
   - Dashboard → lazy-load charts
   - Map pages → lazy-load map SDKs
   - Editor pages → lazy-load rich-text editors
6. Writes report to `.hermes/memory/bundle-split/latest-report.json`
7. On PR failure: comments on PR with top warnings
8. On main push failure: opens GitHub issue labeled `bundle-split`, `performance`, `autonomous`
9. Baseline stored in `baseline-stats.json`; history rolled 90 days

## Configuration

| Threshold | Default | Description |
|-----------|---------|-------------|
| `INITIAL_CHUNK_MAX_KB` | 300 | Max gzipped size for initial chunk (LCP-critical) |
| `ROUTE_CHUNK_MAX_KB` | 250 | Max gzipped size for any route chunk |
| `HEAVY_MODULE_KB` | 50 | Module size threshold considered worth splitting |

Edit `automation/bundle-split-analyzer.cjs` to adjust thresholds.

## Tuning & Customization

### Adding custom route patterns

Modify `generateRecommendations()` in `automation/bundle-split-analyzer.cjs`:

```js
if (route.includes('your-pattern')) {
  recs.push('Your custom advice. Suggest lazy-loading XYZ.');
}
```

### Per-route thresholds

Extend the route report object to include per-route overrides read from a config file.

### Making failures blocking on PR

Edit `.github/workflows/bundle-split-analyzer.yml` — the `Check for warnings` step already exits non-zero on warnings. To make the job fail the PR, remove `continue-on-error: true` from the analyzer step and let the exit code propagate.
