# Autonomous Lazy-Loading Route Slicer

**Status:** ✅ Active  
**Triggers:** Weekly Sunday 04:00 UTC (dry-run analysis), manual dispatch (optional apply)  
**Output:** Proposal diffs under `.hermes/memory/route-slicer/proposals/`; optional PR creation  
**Telegram alerts:** On proposals generated or errors

---

## Problem

Large route-specific chunks increase initial page load time. Components like charts, maps, rich editors, and admin panels are often bundled in the initial JavaScript even though users may never visit those routes or interact with those features. Manual code splitting with `dynamic()` imports is time-consuming and easy to forget.

## Solution

Automated route slicer that:
- Consumes bundle analyzer statistics (#43) to identify heavy modules per route
- Parses `app/` directory route structure and page files
- Detects non-critical, heavy imports
- Proposes `dynamic(() => import(...))` wrappers to defer loading
- Optionally creates a PR with transformations (requires manual approval or explicit apply input)

---

## How It Works

1. **Bundle stats input**: Reads `.next/stats/*.json` generated by `ANALYZE=true` builds (via #43)
2. **Route discovery**: Walks `app/` directory to find all `page.tsx`/`page.ts` files (route segments)
3. **Heavy module detection**: Identifies imported modules whose gzipped size >80KB (configurable)
4. **Transformation proposal**: Generates new page file with `dynamic()` imports injected
5. **Savings estimation**: Computes estimated bytes saved per route (module sizes minus dynamic overhead)
6. **PR creation** (optional): If `apply=true` in manual dispatch, opens PR with changes
7. **Telegram notification**: Alerts when proposals are ready

---

## Configuration

Edit thresholds in `automation/route-slicer.cjs`:

```js
const HEAVY_MODULE_THRESHOLD_KB = 80; // modules larger than this are candidates
const MIN_SAVINGS_KB = 40;           // only propose if estimated saving exceeds this
const MAX_DYNAMIC_PER_ROUTE = 3;     // limit to avoid over-splitting
```

## Usage

### Automatic weekly analysis (dry-run)

Workflow runs every Sunday 04:00 UTC. It:
- Builds the app to generate fresh bundle stats
- Runs route slicer in analysis mode
- Uploads proposal artifacts to workflow
- Sends Telegram notification if proposals found

No code changes are applied automatically. Review proposals at:
```
.hermes/memory/route-slicer/proposals/<timestamp>/
```

### Manual apply (creates PR)

Dispatch the workflow with `apply = true`:
```bash
gh workflow run route-slicer.yml -f apply=true
```

This will:
- Re-run analysis
- Create a PR from `route-slicer-automation-<run_number>` branch
- PR includes transformed files with `dynamic()` imports
- Add label `autonomous, performance, code-split`

**Before merging**, review the changes to ensure:
- Lazy-loaded components are truly non-critical (above-the-fold content should remain synchronous)
- Loading fallbacks (`<p>Loading...</p>`) are appropriate (can customize in script)
- SSR compatibility: `dynamic()` with `{ ssr: false }` may be needed for browser-only components (future enhancement)

## Proposal Structure

```
.hermes/memory/route-slicer/proposals/<timestamp>/
├── summary.json      # { routes: [{ path, savedKB, heavyImports }] }
├── app/
│   └── <route>/
│       └── page.tsx  # transformed version with dynamic() imports
```

`summary.json` example:
```json
{
  "timestamp": "2026-05-12T04:00:00.000Z",
  "routes": [
    {
      "path": "/admin",
      "savedKB": 120,
      "heavyImports": [
        "recharts",
        "react-map-gl",
        "monaco-editor"
      ]
    }
  ]
}
```

## Safety

- **No automatic application**: Dry-run by default; PR requires manual merge (or explicit apply flag)
- **Conservative thresholds**: Only modules >80KB considered; minimum savings 40KB
- **Limited per route**: Max 3 dynamic imports per route prevents fragmentation
- **SSR-aware**: Uses Next.js `dynamic()`; current implementation uses basic loading placeholder (customizable)
- **Atomic review**: Entire route transformation reviewed as a unit; easy to revert

## Troubleshooting

**No bundle stats found:**
Run `ANALYZE=true npm run build` to generate stats in `.next/stats/`. #43's analyzer expects this directory.

**Too many proposals / false positives:**
Increase `HEAVY_MODULE_THRESHOLD_KB` or `MIN_SAVINGS_KB`.

**Transformation incorrect:**
Current implementation uses naive text insertion for dynamic imports. For complex codebases, enhance `generateTransformation()` to use full Babel AST traversal and JSX preservation.

**SSR issues after merge:**
Some components require `ssr: false` option: `dynamic(() => import('module'), { ssr: false, loading: ... })`. Add this to the wrapper template in the script.
