## ✅ Guardrail #34 Deployed — Field Performance & Real User Web Vitals Monitor

**Commit:** `e0b7fd8` pushed to `main`

**What it does:**
- Client-side collector measures LCP, FID, CLS, FCP, TTFB on every pageview using `web-vitals` library
- POSTs to `/api/field-performance` endpoint via `sendBeacon` (reliable even on page unload)
- API stores raw submissions in `.hermes/memory/field-performance/raw/YYYY-MM-DD.jsonl`
- Daily aggregator runs 11:00 UTC:
  - Computes p75/p90 per page/metric
  - Compares to 7-day moving average baseline
  - Detects regressions (>15% worsening)
  - Calculates overall site score (0–100 weighted average)
  - Sends Telegram summary (top 10 worst pages, trends, score)
  - Creates GitHub issue if >5% pages critical OR >20% regression on any metric
- Thresholds: LCP >2.5s warning/>4s critical, FID >100ms/>300ms, CLS >0.1/>0.25, FCP >1.8s, TTFB >800ms
- State retention: 90 days aggregates/history; 30 days raw data
- Privacy: URL query params stripped; no PII collected

## ✅ Guardrail #36 Deployed — Performance Budget Enforcement (CI Gate)

**Commit:** `d364d29` pushed to `main`

**What it does:**
- Runs Lighthouse on every PR's built site (served locally via `npx serve out -p 3000`)
- Extracts Core Web Vitals: LCP, FID, CLS, FCP, TTFB + overall performance score
- Compares metrics against baseline stored in `.hermes/memory/performance-budget/baseline.json`
- Fails CI if:
  - Any metric exceeds **hard limit**: LCP >4s, FID >300ms, CLS >0.25, FCP >3s, TTFB >2s
  - OR any metric regresses >10% relative to 7-day baseline
- Updates baseline automatically on successful `main` branch runs (when `IS_MAIN_BRANCH=true`)
- Stores 90-day history in `.hermes/memory/performance-budget/history.json`
- Workflow: `.github/workflows/performance-budget.yml` — triggers on PR to `main`; also manual dispatch
- Exit code 0 = pass; 1 = fail (blocks merge if branch protection requires check)
- Dependencies: `lighthouse`, `yargs` added to devDependencies

**Files added:**
- `automation/performance-budget-enforcer.cjs`
- `.github/workflows/performance-budget.yml`
- `docs/PERFORMANCE-BUDGET.md`

**Why it matters:**
- **Prevents regressions before they ship:** Existing monitors alert after deploy; this gate stops bad performance from being merged.
- **Immediate feedback:** Developers see performance failures in CI before code reaches users.
- **Complements existing guards:** Lighthouse daily monitor provides trends; Field Performance provides real-user data; Budget Enforcement enforces standards at PR time.
- **Cost:** $0 — uses GitHub Actions minutes and local lighthouse.

**Operation:**
- On PR: baseline is read-only; run compares current metrics to stored baseline; failure prints detailed metric diffs.
- On main (manual or future scheduled run): baseline is updated to the latest good metrics, refreshing standards as site improves.
- Hard limits ensure metrics never cross Google's "poor" thresholds that hurt SEO.
- Regression threshold (10%) catches meaningful degradation while allowing natural variance.

**Next steps:**
- Baseline will be populated after first successful main-branch run (manual trigger or integrated into daily deploy pipeline)
- Consider adding route-specific budgets in future (different pages have different realistic LCP targets)
- Optional: enrich PR comment with per-metric suggestions (image optimization, reduce JS, etc.)
